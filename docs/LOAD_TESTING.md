# Нагрузочное тестирование LoyaltyLoop

Этот документ описывает процесс проведения нагрузочного тестирования сервера LoyaltyLoop с использованием инструмента **Locust**.

## Цель
Проверить стабильность системы при высокой нагрузке, эмулируя одновременную работу множества кассиров (сканирование QR, проведение транзакций).

## Предварительные требования
1.  **Python 3.8+**
2.  **Locust**: Установите через pip:
    ```bash
    pip install locust
    ```
3.  **Тестовые данные**:
    *   Создайте тестового Кассира и получите его `access_token` (можно взять из LocalStorage браузера или логов при входе).
    *   Создайте тестового Клиента и узнайте его `id` и `qrSecret` (можно посмотреть в БД в таблице `users`).
    *   Узнайте `id` торговой точки (`trading_points`).

## Запуск теста

1.  Откройте терминал в папке проекта.
2.  Запустите Locust, передав переменные окружения с тестовыми данными:

```bash
# Пример (замените значения на реальные!)
export CASHIER_TOKEN="eyJhbGciOiJIUzI1NiJ9..."
export POINT_ID="point-uuid-123"
export TEST_USER_ID="user-uuid-456"
export TEST_QR_SECRET="user-qr-secret-789"

locust -f tests/load_test/locustfile.py --host https://loyaltyloop.up.railway.app
```

3.  Откройте браузер по адресу `http://localhost:8089`.
4.  Введите параметры теста:
    *   **Number of users**: Количество одновременных кассиров (начните с 10, затем 100, 1000).
    *   **Spawn rate**: Скорость добавления пользователей (например, 5 в секунду).
5.  Нажмите "Start Swarming".

## Сценарий теста
Скрипт эмулирует поведение кассира:
1.  **Сканирование QR (`/terminal/scan`)**: Кассир сканирует QR-код клиента. Генерируется валидная подпись HMAC-SHA256.
2.  **Проведение транзакции (`/terminal/process`)**: Кассир вводит сумму покупки и подтверждает операцию.

## Анализ результатов
*   **RPS (Requests Per Second)**: Сколько операций в секунду выдерживает сервер.
*   **Failures**: Процент ошибок (500, 502, timeouts). Должен быть 0%.
*   **Response Time (Median / 95%ile)**: Время ответа. Для комфортной работы кассира должно быть < 500мс.

## Тестирование на Railway
Чтобы протестировать production-окружение в Railway:
1.  Убедитесь, что база данных имеет достаточный пул соединений (`HikariCP` настройки).
2.  Следите за метриками CPU и RAM в панели Railway во время теста.
3.  Если возникают ошибки подключения к БД, возможно, стоит увеличить тарифный план БД или оптимизировать пул.

## Рекомендации по оптимизации
Если тест показывает низкую производительность:
1.  **База данных**: Проверьте индексы в таблицах `transactions_history` и `loyalty_cards`.
2.  **Пул соединений**: Увеличьте `maximumPoolSize` в `DatabaseFactory.kt` (сейчас дефолт, для нагрузки нужно 10-20+).
3.  **Логирование**: Отключите подробное логирование в продакшене, так как запись в консоль может блокировать потоки.
