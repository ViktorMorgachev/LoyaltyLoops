# Авторизация и Регистрация через Telegram (Under the Hood)

Этот документ описывает техническую реализацию процесса входа и регистрации пользователей через Telegram бот.

## Общая концепция

Авторизация построена на механизме **Cross-Device Authentication** (аналогично входу в WhatsApp Web или Telegram Desktop по QR-коду). 
Пользователь инициирует вход на клиенте (Веб или Мобильное приложение), получает уникальную ссылку на бот, и подтверждает свою личность внутри Telegram. Сервер связывает сессию клиента с Telegram-аккаунтом пользователя.

---

## Компоненты системы

1.  **Frontend (Web/Mobile)**: Инициатор сессии. Отображает QR-код/Кнопку и опрашивает сервер о статусе входа.
2.  **Backend (Ktor)**:
    *   `TelegramAuthService`: Служба, которая "слушает" события от Telegram (Polling) и управляет бизнес-логикой бота.
    *   `AuthSessionRepository`: Управляет временными сессиями входа.
    *   `UserRepository`: Управляет пользователями и их привязкой к Telegram ID.
3.  **Database**:
    *   `auth_sessions`: Таблица для хранения временных сессий (UUID, статус, связь с Telegram).
    *   `users`: Таблица пользователей, теперь содержит поле `telegram_id`.
4.  **Telegram Bot**: Внешний интерфейс для подтверждения номера телефона.

---

## Подробный алгоритм (Flow)

### 1. Инициация сессии (Frontend -> Backend)
1.  Пользователь нажимает "Войти через Telegram".
2.  Клиент делает запрос `POST /api/auth/telegram/start`.
3.  Сервер:
    *   Генерирует уникальный `UUID` сессии.
    *   Создает запись в `auth_sessions` со статусом `PENDING`.
    *   Возвращает `UUID` и имя бота (`botUsername`) клиенту.
4.  Клиент формирует ссылку вида: `https://t.me/{botUsername}?start=login_{UUID}`.
    *   На Вебе: Генерирует QR-код из этой ссылки.
    *   На Мобайле: Открывает приложение Telegram по этой ссылке (Deep Link).
5.  Клиент начинает опрашивать сервер `GET /api/auth/telegram/status/{UUID}` каждые N секунд.

### 2. Взаимодействие с Ботом (User -> Telegram -> Backend)
1.  Пользователь переходит в бот и нажимает **Start** (или ссылка сама триггерит команду `/start login_{UUID}`).
2.  `TelegramAuthService` получает обновление (Update) с текстом `/start login_{UUID}`.
3.  **Поиск сессии**: Сервер находит сессию по `UUID` в `auth_sessions`.
4.  **Проверка привязки**:
    *   Сервер проверяет, есть ли в таблице `users` пользователь с таким `telegram_id` (Chat ID).
    *   **Сценарий А (Пользователь уже привязан)**:
        *   Сервер сразу обновляет сессию: статус `CONFIRMED`, `user_id` = ID найденного пользователя.
        *   Бот отправляет сообщение: "Вы успешно вошли!".
    *   **Сценарий Б (Пользователь новый или не привязан)**:
        *   Сервер временно "линкует" этот `chat_id` к сессии (записывает `telegram_id` в `auth_sessions`), но статус остается `PENDING`.
        *   Бот отправляет сообщение с просьбой поделиться контактом и специальную клавиатуру: `[📱 Поделиться контактом]`.

### 3. Верификация контакта (User -> Telegram -> Backend)
*Этот шаг выполняется только для Сценария Б.*

1.  Пользователь нажимает кнопку "Поделиться контактом".
2.  Telegram отправляет боту объект `Contact`, содержащий `phone_number` и `user_id`.
3.  `TelegramAuthService` проверяет безопасность:
    *   `contact.user_id` должен совпадать с `message.chat.id` (чтобы нельзя было переслать чужой контакт).
4.  **Поиск пользователя**:
    *   Сервер ищет пользователя в БД по номеру телефона (`phoneNumber`).
    *   **Если найден**:
        *   Сервер сохраняет `telegram_id` в запись пользователя (линковка навсегда).
        *   Обновляет сессию: статус `CONFIRMED`, `user_id` = ID пользователя.
    *   **Если НЕ найден (Регистрация)**:
        *   Сервер создает нового пользователя (`UserDto`) с данными из Telegram (Телефон, Имя, Фамилия).
        *   Сохраняет `telegram_id`.
        *   Обновляет сессию: статус `CONFIRMED`, `user_id` = ID нового пользователя.
5.  Бот отвечает: "Телефон подтвержден! Вы вошли."

### 4. Завершение входа (Frontend <-> Backend)
1.  Очередной запрос клиента `GET /api/auth/telegram/status/{UUID}` попадает на сервер.
2.  Сервер видит, что статус сессии `CONFIRMED` и есть `user_id`.
3.  Сервер:
    *   Генерирует JWT токены (Access + Refresh).
    *   Возвращает токены и профиль пользователя клиенту.
4.  Клиент сохраняет токены и переадресует пользователя в приложение.

## Структура БД

### Таблица `auth_sessions`
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | VARCHAR | UUID сессии (Primary Key) |
| `status` | VARCHAR | `PENDING` (ждет), `CONFIRMED` (успех), `EXPIRED` (истекла) |
| `telegram_id` | LONG | ID чата в Telegram (заполняется при старте бота) |
| `phone` | VARCHAR | Телефон (заполняется после шеринга контакта) |
| `user_id` | VARCHAR | ID пользователя в нашей системе (заполняется при успехе) |
| `created_at` | LONG | Время создания |
| `expires_at` | LONG | Время истечения жизни сессии |

### Таблица `users` (изменения)
Добавлено поле:
*   `telegram_id` (Long, Unique, Nullable) — для быстрой авторизации без повторного запроса контакта.

---

## Безопасность и Ограничения
1.  **TTL Сессии**: Сессия живет ограниченное время (например, 2-5 минут), настраивается в конфиге.
2.  **Проверка Контакта**: Бот принимает только "свои" контакты (поле `user_id` в контакте должно совпадать с отправителем), чтобы предотвратить спуфинг.
3.  **Привязка**: Один Telegram аккаунт привязывается к одному Пользователю LoyaltyLoop.
