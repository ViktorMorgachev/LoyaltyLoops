# Screen Documentation Guide

> **Статус:** актуализирован апрель 2026. Единый и единственный гайд по screen-driven документации. Изменения вносятся редко — это foundation-документ.

## Назначение

Этот документ — единственный источник правил для screen-driven документации в LoyaltyLoops.

Каждый экран описывается **парой документов**:

- `SCREEN_NAME.md` — продуктовый экранный документ
- `SCREEN_NAME_SPEC.md` — техническая спецификация экрана

Сюда же вшиты:
- обязательные блоки SPEC-документа (раздел [Структура SPEC](#структура-spec));
- стандарт ASCII-визуализации (раздел [Визуальная карта экрана](#визуальная-карта-экрана));
- шаблон нового SPEC (раздел [Шаблон SPEC](#шаблон-spec)).

> Раньше эти три темы были разнесены по `SCREEN_DOCUMENTATION_GUIDE.md` + `SCREEN_SPEC_STRUCTURE_STANDARD.md` + `SCREEN_SPEC_TEMPLATE.md`. Теперь — один файл, чтобы не сверять три источника при каждом новом экране.

---

## Основное правило

Сначала фиксируется экран и его spec, потом создаётся или обновляется task в `docs/tasks/`.

Порядок работы:

1. создать или обновить `SCREEN_NAME.md`
2. создать или обновить `SCREEN_NAME_SPEC.md`
3. создать или обновить task-документ этапа
4. после этого переходить к реализации

---

## Что хранить в `SCREEN_NAME.md`

`SCREEN_NAME.md` — это продуктовый экранный документ.

Он отвечает на вопросы:

- зачем существует экран
- когда экран показывается
- какой пользовательский смысл у экрана
- какие основные блоки есть на экране
- какие CTA есть у экрана
- какой экранный результат ожидается

Это не место для длинных переходных таблиц, runtime-деталей и backend-контрактов.

---

## Что хранить в `SCREEN_NAME_SPEC.md`

`SCREEN_NAME_SPEC.md` — это source of truth для реализации.

В spec фиксируются:

- route
- entry rules
- redirect rules
- required and optional data
- states
- transitions
- edge cases
- analytics reference
- error handling
- acceptance criteria

---

## Именование файлов

Используем верхний регистр и явные screen ids:

- `WELCOME_SCREEN.md`
- `WELCOME_SCREEN_SPEC.md`
- `PARTNER_DASHBOARD_SCREEN.md`
- `PARTNER_DASHBOARD_SCREEN_SPEC.md`
- `REGISTRATION_REQUIRED_SCREEN.md`
- `REGISTRATION_REQUIRED_SCREEN_SPEC.md`

---

## Структура директорий

Экранные документы хранятся по модулям:

- `docs/screens/core/`
- `docs/screens/partner/`
- `docs/screens/shared/`
- `docs/screens/webadmin/`

Step-задачи хранятся отдельно:

- `docs/tasks/` — активные планы этапов
- `docs/archive/tasks/` — архив сданных STEP'ов

---

## Правило для prompt-экранов

Prompt-экраны должны быть названы по продуктовой причине показа, а не по абстрактному prompt-термину.

Используем:

- `REGISTRATION_REQUIRED_SCREEN`
- `SUPPORTER_REQUIRED_SCREEN`
- `REGISTRATION_SUPPORTER_REQUIRED_SCREEN`

Не используем в новой документации:

- `REGISTRATION_PROMPT`
- `SUPPORTER_PROMPT`
- `REGISTRATION_SUPPORTER_PROMPT`

---

## Правило для соответствия коду

Если экран уже реализован, screen doc и screen spec должны быть подбиты под текущее поведение кода.

Нельзя документировать несуществующую реализацию как уже готовую.

Если в коде есть ограничение, оно должно быть явно зафиксировано в spec.

---

## Что делать, если экран ещё не реализован

Если экран ещё не реализован:

- в `SCREEN_NAME.md` фиксируется продуктовая цель экрана;
- в `SCREEN_NAME_SPEC.md` фиксируется целевое поведение;
- в task-документе этапа явно указывается scope;
- в spec обязательно помечается, что экран ещё не реализован.

---

## Что не делать

- не смешивать screen doc и task в одном файле;
- не смешивать продуктовый UX и backend-контракт без явного разделения;
- не оставлять старые названия экранов после переименования;
- не придумывать runtime-логику, если она не подтверждена кодом или отдельной продуктовой фиксацией.

---

## Правило для аналитики

Имена событий, payload и правила отправки аналитики описываются только в `ANALYTICS_EVENTS.md`.

Screen spec может содержать только ссылку на `ANALYTICS_EVENTS.md`, но не должен дублировать event names и payload.

---

## Структура SPEC

Каждый `*_SCREEN_SPEC.md` должен содержать обязательные блоки в фиксированном порядке. Если блок не применим, это указывается явно («Не используется», «нет»).

### 1. Общая информация
Минимум:
- Screen ID
- Route
- Module
- Status (`draft / approved / updated`)
- Implementation status (`not_implemented / partially_implemented / implemented`)

### Визуальная карта экрана
Перед всеми техническими секциями. Подробности — раздел [Визуальная карта экрана](#визуальная-карта-экрана) ниже.

### 2. Entry Rules
- откуда пользователь попадает
- при каких условиях экран допустим
- можно ли открыть экран напрямую
- требуется navigation state (`yes / no`)

### 3. Redirect Rules
- когда экран не должен показываться
- куда уводить пользователя
- что делать при потерянном navigation state

### 4. Data Contract
- required data
- optional data
- source of data (endpoint, props, location.state)
- можно ли показать экран без backend (`yes / no`)

### 5. States
По возможности описываются: `default`, `loading`, `empty`, `error`, `disabled`, `success`. Если состояние не используется — указывается явно.

### 6. Transitions
Перечисляются:
- пользовательские действия
- переходы по CTA
- back behavior
- route transitions

### 7. Edge Cases
Фиксируются:
- потерянный `location.state`
- невалидный route param
- отсутствие данных
- частично заполненный backend response
- повторный вход на экран

### 8. Analytics Reference
Только ссылка на `ANALYTICS_EVENTS.md`. Event names, payload и правила отправки событий не должны дублироваться в screen spec.

### 9. Error Handling
- откуда берётся ошибка
- как она показывается
- есть ли retry
- куда возвращать пользователя

### 10. Acceptance Criteria
- что считается завершённой реализацией экрана
- какие состояния и переходы должны работать

### Дополнительные блоки (опционально)

При необходимости можно добавлять:
- UI Notes
- Copy Notes
- Accessibility Notes
- Backend Notes
- Future Extension
- Non-goals

## Визуальная карта экрана

### Зачем

ASCII-визуализация экрана — **стандартный способ описания layout в `SCREEN_NAME_SPEC.md`**. Она даёт разработчику однозначное представление о структуре экрана без мокапов и без необходимости открывать Figma. Живёт рядом с техническими деталями — в том же документе.

### Требования

Каждый `SCREEN_NAME_SPEC.md` должен содержать раздел **«Визуальная карта экрана»** — между «1. Общая информация» и «2. Entry Rules».

Каждое значимое состояние экрана (`loading`, `default`, `locked`, `error`, `empty`, `video-only` и т.д.) описывается **отдельным ASCII-блоком**.

### Синтаксис

Блок рисуется в ` ``` ` без языка (plain text). Используются символы псевдографики:

```
┌ ─ ┐   — границы блоков
│   │   — вертикальные стороны
└ ─ ┘   — нижние углы
├ ─ ┤   — горизонтальный разделитель внутри блока
```

Состояния элементов передаются символами заполнения:

| Символ | Смысл |
|---|---|
| `░` | skeleton / заглушка при загрузке |
| `▒` | размытый / скрытый контент (blur) |
| `▓` | gradient fade — контент плавно исчезает |

Аннотации пишутся справа от блока через `←`:

```
│  Заголовок карточки                     │  ← font-size: 20px, font-weight: 700
│  (до 2–3 строк)                         │
```

Если нужно указать CSS-свойство или поведение — пишется рядом с аннотацией:

```
│  ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│  │  ← filter: blur(8px) brightness(0.6)
```

### Пример — состояние loading

```
┌─────────────────────────────────────────┐
│ ←                                       │
├─────────────────────────────────────────┤
│                                         │
│  ┌───────────────────────────────────┐  │
│  │░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░│  │  ← cover skeleton
│  └───────────────────────────────────┘  │
│                                         │
│  ░░░░░░░░░░░░░░░░░░░░░░░░░░░           │  ← title skeleton
│  ░░░░░░░░░░░░░░░░░░░░                  │
│                                         │
│  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░   │  ← body skeleton
│  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░   │
│  ░░░░░░░░░░░░░░░░░░░░░               │
│                                         │
└─────────────────────────────────────────┘
```

### Пример — locked / paywall состояние

```
│  ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒│  │  ← cover: blur(8px) brightness(0.6)
│  начало текста видно...               │  ← bodyLockedPreview: max-height 220px
│  плавное исчезновение вниз            │
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  │  ← mask-image gradient (black → transparent)
│  ┌─────────────────────────────────┐  │
│  │  🔒  Откроется при поддержке    │  │  ← paywallCta, margin-top: 0
│  └─────────────────────────────────┘  │
```

### Правила оформления

- Блок всегда начинается с `┌─…─┐` и заканчивается `└─…─┘`
- Аннотации выравниваются по правому краю через два пробела
- Внутри блока используются вложенные блоки (`┌…┐` внутри `│…│`) для карточек, изображений и секций
- Название состояния пишется как заголовок `###` над блоком
- После ASCII-блока допустим краткий текст — только если что-то требует пояснения, которое нельзя уместить в аннотацию

### Где не нужна ASCII-визуализация

- В `SCREEN_NAME.md` (продуктовый документ — только текст)
- В task-документах этапов (там описывается scope, не визуал)
- Для trivially простых экранов без состояний (например, экран-заглушка с одной кнопкой)
