# План анализа нагрузки и многопоточности

## 1. Текущее состояние (Предварительный анализ)
Система построена на **Ktor (Netty)** и **Exposed (JDBC)**.
*   **Ktor** использует корутины, что обеспечивает высокую эффективность обработки I/O операций (тысячи одновременных соединений).
*   **Exposed** использует блокирующий JDBC драйвер (обычно), поэтому database calls обернуты в `dbQuery { ... }` (который использует `Dispatchers.IO`).

### Потенциальные узкие места (Bottlenecks):
1.  **Database Connection Pool**: Если пул соединений (HikariCP) слишком мал, запросы будут вставать в очередь.
2.  **Транзакции (`TransactionService.processTransaction`)**:
    *   Проверки (баланс, статус) и обновление происходят в рамках одной бизнес-логики, но не всегда в рамках одной ACID транзакции базы данных (нужно проверить `dbQuery` границы).
    *   **Race Condition**: Если два кассира одновременно сканируют одну карту, возможна ситуация "Double Spending" (двойное списание), если уровень изоляции транзакций недостаточен или логика не атомарна.
3.  **Аналитика**: Агрегация данных (`RatingRepository`, `TransactionService`) происходит **в памяти** (`query.toList()`, `groupBy`), а не в базе данных.
    *   *Риск*: При росте количества транзакций до 100к+ это "уронит" сервер по памяти (OOM) или CPU.
4.  **Синхронные вызовы внешних API**: (Telegram, SMS) могут замедлять ответы, если не вынесены в фоновые Job-ы.

## 2. План нагрузочного тестирования (Load Testing)

Мы будем использовать **k6** или **JMeter** для симуляции нагрузки.

### Сценарии тестов:
1.  **Auth Spike**: 1000 пользователей одновременно пытаются войти (SMS/Telegram). Проверка работы Rate Limiter и пула соединений.
2.  **Transaction Flood**: 100 кассиров проводят транзакции каждые 2 секунды. Проверка на Deadlocks и Race Conditions.
3.  **Analytics Heavy**: 10 партнеров одновременно запрашивают "Отчет за год". Проверка потребления памяти.

### Метрики успеха:
*   P95 Latency < 500ms для транзакций.
*   0% ошибок (500/503) при нагрузке x10 от текущей.
*   Корректные балансы (отсутствие отрицательных значений или расхождений).

## 3. План рефакторинга (Multithreading Safety)

1.  **Аудит транзакций**:
    *   Перенести логику `check -> calculate -> update` внутрь **одной** SQL транзакции (`TransactionManager.current()....`).
    *   Использовать `SELECT ... FOR UPDATE` для блокировки строки карты при списании.
2.  **Оптимизация аналитики**:
    *   Переписать in-memory группировку (`groupBy`, `sumOf`) на SQL-агрегации (`GROUP BY`, `SUM()`). Это критически важно для масштабируемости.
3.  **Кэширование**:
    *   Внедрить Redis для хранения сессий и часто запрашиваемых данных (настройки партнера), чтобы разгрузить БД.

### 1. Текущее состояние (Анализ)

Все верно, но добавь внимание к следующим деталям:

*   **Dispatchers.IO vs HikariCP**:
    *   *Проблема:* Размер пула потоков `Dispatchers.IO` (по умолчанию 64 или кол-во ядер) может быть больше, чем размер пула соединений HikariCP (по умолчанию часто 10).
    *   *Риск:* Корутины будут блокироваться не на I/O, а в ожидании свободного слота в пуле БД.
    *   *Решение:* Размер пула HikariCP должен быть сопоставим с конкурентностью, которую вы ожидаете от `dbQuery`.
*   **Race Conditions (Состояние гонки)**:
    *   Ты верно заметил про Double Spending. В Exposed это часто происходит, когда делают:
        ```kotlin
        val balance = account.balance // SELECT
        if (balance >= amount) {
            account.balance = balance - amount // UPDATE
        }
        ```
    *   Между SELECT и UPDATE вклинивается другой поток.

### 3. Рефакторинг (Multithreading Safety)

Самый важный раздел. Вот конкретные правки:

#### А. Аудит транзакций (Блокировки)
Использование `SELECT ... FOR UPDATE` — это **пессимистическая блокировка**. Это надежно, но убивает производительность (concurrency).

*   **Вариант 1 (Пессимистический, твой выбор):**
    В Exposed это делается через `.forUpdate()`.
    *Плюс:* Гарантирует целостность.
    *Минус:* Если транзакция долгая (например, внутри есть запрос к внешнему API), вся таблица/строка висит. **Никогда не делайте HTTP-запросы внутри SQL-транзакции с блокировкой!**

*   **Вариант 2 (Оптимистический):**
    Добавить колонку `version` в таблицу счета.
    При обновлении делать:
    `UPDATE accounts SET balance = X, version = version + 1 WHERE id = Y AND version = old_version`
    Если количество обновленных строк = 0, значит, кто-то успел изменить баланс раньше -> кидаем исключение и просим клиента повторить.

**Рекомендация:** Для финансового биллинга `FOR UPDATE` (пессимистический) подходит лучше, но транзакции должны быть **максимально короткими**.

#### Б. Оптимизация аналитики
Перенос вычислений в SQL (`GROUP BY`, `SUM`) — это 100% верно.
*   **Важное дополнение:** SQL агрегации будут медленными без **индексов**.
    *   Убедись, что в плане есть пункт: "Проанализировать `EXPLAIN ANALYZE` тяжелых запросов и добавить индексы (например, составной индекс `(partner_id, created_at)`)".

#### В. Внешние API (Async)
Ты упомянул "фоновые Job-ы".
*   *Риск:* Если использовать просто `CoroutineScope.launch` внутри памяти приложения, то при перезагрузке сервера (или крэше OOM) **задачи потеряются**. СМС не уйдет, чек в Телеграм не придет.
*   *Решение:* Для надежности нужен **Persistent Queue** (хотя бы таблица в БД `task_queue` или Redis/RabbitMQ), откуда воркеры берут задачи. Если задача упала — ретрай.

Можешь приступать к написанию тестов на Race Condition. Для этого в Kotlin удобно использовать `Dispatchers.Default` с `launch` в цикле:

```kotlin
// Пример теста на Race Condition
@Test
fun `test race condition`() = runBlocking {
    val startBalance = 1000
    // ... init db ...

    val jobs = List(100) {
        launch(Dispatchers.Default) {
            service.processTransaction(amount = 10)
        }
    }
    jobs.joinAll()

    val finalBalance = repository.getBalance()
    // Если 0, то все ок. Если > 0, значит Race Condition сработал.
    assertEquals(0, finalBalance)
}
```
