Чтобы PostgreSQL работал как швейцарские часы и всегда хранил/отдавал время в UTC, нужно настроить его на двух уровнях: **Инфраструктура** (Docker/Railway) и **Сама База Данных**.

Вот пошаговая инструкция.

---

### Вариант 1. Если ты используешь Docker Compose (Локально)

Самый надежный способ — сказать Postgres при запуске, в какой зоне жить.

Добавь эти строки в твой `docker-compose.yml` в секцию базы данных:

```yaml
services:
  postgres:
    image: postgres:15-alpine
    # 1. Заставляем сам процесс Postgres работать в UTC
    command: postgres -c timezone=UTC
    
    # 2. Заставляем операционную систему контейнера (Linux) жить в UTC
    # (Это влияет на системные логи и cron внутри контейнера)
    environment:
      - TZ=UTC
      - PGTZ=UTC
```

---

### Вариант 2. Если База уже создана (Railway / Production)

На облачных хостингах (Railway, AWS RDS, DigitalOcean) у тебя часто нет доступа к файлу конфигурации `postgresql.conf` или параметрам запуска Docker.

В этом случае используй **SQL-команду**. Она сохраняет настройку навсегда для конкретной базы данных.

1.  Подключись к базе через терминал или клиент (IntelliJ IDEA, pgAdmin, DBeaver).
2.  Выполни запрос:

```sql
-- Заменяем "loyalty_db" на имя твоей базы данных
ALTER DATABASE "loyalty_db" SET timezone TO 'UTC';
```

3.  **Важно:** Чтобы изменения вступили в силу для *новых* подключений, нужно перезапустить приложение (Ktor) или переподключиться. Саму базу перезагружать не обязательно.

---

### Вариант 3. "Железная гарантия" в коде (HikariCP)

Даже если админ сервера случайно поменяет время на сервере на "Asia/Bishkek", твое приложение может принудительно требовать UTC при каждом подключении.

В коде, где ты настраиваешь `HikariConfig` (подключение к БД), добавь это:

```kotlin
val config = HikariConfig().apply {
    jdbcUrl = "jdbc:postgresql://..."
    // ... логин, пароль ...

    // ВАЖНО: Выполнять эту команду при каждом создании соединения
    connectionInitSql = "SET TIME ZONE 'UTC'"
}
```

Это самый безопасный метод ("Пояс и подтяжки"), он работает всегда и везде.

---

### Как проверить, что всё работает?

Зайди в консоль базы данных и выполни:

```sql
SHOW timezone;
```

*   Если видишь **`UTC`** — всё отлично.
*   Если видишь `Asia/Bishkek` или `+06` — настройки не применились.

### И важный нюанс про типы данных

В Postgres есть два типа timestamp:
1.  `TIMESTAMP` (timestamp without time zone) — просто сохраняет "2024-12-20 10:00". Ему плевать на таймзону сервера. Если ты записал 10:00, он отдаст 10:00.
2.  `TIMESTAMPTZ` (timestamp with time zone) — при сохранении он конвертирует время в UTC (используя настройку сервера), а при чтении конвертирует обратно.

**Рекомендация:**
Так как мы используем Exposed и `datetime` (`LocalDateTime`), лучше всего настроить сервер на UTC (как описано выше) и использовать тип `TIMESTAMP` (или `datetime` в Exposed), всегда передавая туда `LocalDateTime.now(ZoneOffset.UTC)`.

В твоем коде (`nowUtc()`) мы это уже сделали, так что проблем не будет. 🚀
