# 🔄 Архитектура и Флоу Системы Лояльности

Этот документ подробно описывает технические процессы (User Flows) системы LoyaltyLoop.

---

## 1. Акторы системы

*   **User (Клиент)**: Пользуется мобильным приложением, копит баллы, ищет заведения.
*   **Cashier (Кассир)**: Сотрудник партнера. Использует приложение в режиме "Терминал" для сканирования QR и проведения оплат.
*   **Partner (Владелец)**: Управляет бизнесом через Web-Admin (настройки лояльности, персонал, статистика).
*   **System (Server)**: Единый источник правды. Обрабатывает транзакции, хранит балансы, рассылает пуши.

---

## 2. Флоу Аутентификации (Auth Flow)

Система использует **Passwordless** вход (OTP по SMS/WhatsApp) + **JWT** (Access/Refresh).

1.  **Вход**:
    *   Клиент вводит номер телефона.
    *   Сервер генерирует 4-значный код и отправляет его (в dev-режиме возвращает в ответе).
    *   Клиент вводит код -> Сервер выдает пару `AccessToken` (1 час) + `RefreshToken` (30 дней).
2.  **Регистрация**:
    *   Если телефона нет в базе, создается новый `User` со случайным `qrSecret` (ключ для генерации динамических QR).
3.  **Ролевая модель**:
    *   При входе сервер возвращает список доступных воркспейсов (бизнесов).
    *   В приложении пользователь выбирает роль:  `CASHIER` или `PARTNER` по умолчанию `CLIENT` после регистрации,

---

## 3. Флоу Транзакции (Transaction Loop)

Самый критичный процесс системы. Весь расчет происходит на сервере.

### Шаг 1: Идентификация (Handshake)
1.  **Клиент**: Открывает QR-код (генерируется локально на основе `qrSecret` + TimeStamp, обновляется каждые 60 сек).
2.  **Кассир**: Сканирует QR камерой устройства.
3.  **Терминал**: Отправляет токен из QR на сервер (`GET /terminal/scan`).
4.  **Сервер**:
    *   Валидирует подпись QR.
    *   Находит пользователя.
    *   Возвращает кассиру: Имя клиента, Текущий баланс в этом заведении, Текущий уровень (Tier).

### Шаг 2: Расчет (Calculation)
1.  **Кассир**: Вводит сумму чека (например, 1000 сом).
2.  **Терминал**: Запрашивает расчет (`POST /terminal/calculate`).
3.  **Сервер**:
    *   Определяет стратегию партнера (Кэшбэк или "Кофейная карта").
    *   Считает, сколько бонусов будет начислено.
    *   Проверяет, сколько бонусов можно списать (учитывая `maxBurnPercentage`).
    *   Возвращает: `potentialEarn`, `maxBurn`, `newTier`.

### Шаг 3: Проведение (Execution)
1.  **Кассир**: Спрашивает клиента "Списываем баллы?". Выбирает действие в UI.
2.  **Терминал**: Отправляет команду (`POST /terminal/transaction`).
3.  **Сервер**:
    *   **Atomic Transaction**: В рамках одной транзакции БД обновляет баланс карты, создает запись в истории, обновляет статистику партнера.
    *   **Level Up**: Если сумма покупок превысила порог, повышает уровень лояльности.
    *   **Real-time**: Отправляет событие через WebSocket клиенту.
    *   **Push**: Отправляет FCM пуш "Вам начислено 50 баллов" (В разработке).

---

## 4. Флоу Real-time (WebSockets)

Используется для мгновенного обновления баланса на экране клиента без "pull-to-refresh".

1.  **Подключение**: Клиент при старте открывает сокет `wss://api.../ws/cards?token=...`.
2.  **События**:
    *   `CARD_UPDATED`: Изменился баланс.
    *   `TRANSACTION`: Прошла покупка (показывает красивую анимацию конфетти).
3.  **Визуализация**: Экран "Кошелек" слушает этот поток и обновляет цифры "на лету".

---

## 5. Стратегии Лояльности

Система поддерживает гибридную логику, настраиваемую партнером.

### A. Tiered Cashback (Уровневая)
Классическая накопительная система.
*   **Логика**: Чем больше клиент тратит за все время (LTV - Lifetime Value), тем выше его % кэшбэка.
*   **Пример**:
    *   0 - 10,000 сом -> **Bronze** (3%)
    *   10,000 - 50,000 сом -> **Silver** (5%)
    *   50,000+ сом -> **Gold** (10%)
*   **Сгорание**: Можно настроить сгорание бонусов, если клиент не приходил X дней.

### B. Visit Counter (Счетчик посещений)
Механика "6-й кофе в подарок".
*   **Логика**: Кэшбэк не начисляется. Считаются только факты оплаты (чеки).
*   **Цель**: Набрать N посещений.
*   **Награда**: При достижении цели кассир видит уведомление "Положен подарок". После выдачи счетчик сбрасывается.

### C. Hybrid (Смешанная)
*   Копим баллы (как в Tiered) + Считаем посещения.
*   Используется для сложной геймификации.

## 6. Картография и Геопоиск

1.  **База данных**: Используется PostGIS (или математический расчет Haversine в коде) для поиска точек в радиусе.
2.  **Кластеризация**: На клиенте (Yandex MapKit) точки объединяются в кластеры.
3.  **Фильтрация**:
    *   По типу бизнеса (Кофейня, Ресторан).
    *   По статусу "Открыто сейчас" (сверка с графиком работы на сервере).
    *   "Только мои карты" (показывает точки, где у юзера уже есть история, для партнера).
