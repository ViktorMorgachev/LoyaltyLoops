# Архитектура B2B CRM (LoyaltyLoop)

## Обзор
Модуль B2B CRM превращает LoyaltyLoop в платформу управления продажами. 
Он позволяет выстраивать иерархию менеджеров, 
Продавать подписки партнерам, 
Управлять жизненным циклом подписок и ограничивать доступ на основе ролей.

## 1. Ролевая модель (RBAC)

### Иерархия
1.  **PLATFORM_SUPER_ADMIN (Владелец)**
    *   Полный доступ ко всему.
    *   Может приглашать `SUPER_MANAGER` и `MANAGER`.
    *   Может увольнять кого угодно.
    *   Может одобрять/отклонять любые заявки.
    *   Видит системные логи.
2.  **PLATFORM_SUPER_MANAGER (Руководитель отдела продаж)**
    *   Может приглашать **только** `MANAGER`.
    *   Не может уволить другого `SUPER_MANAGER` или Владельца.
    *   Видит все заявки и статистику подчиненных.
    *   Может одобрять/отклонять заявки.
    *   Видит логи (ограниченно или полностью).
3.  **PLATFORM_MANAGER (Менеджер по продажам)**
    *   Может создавать заявки (Активация точки, Блокировка партнера).
    *   Видит ТОЛЬКО свои заявки.
    *   НЕТ доступа к системным логам.
    *   Доступ к деталям Партнера только на чтение (кроме создания заявок).

## 2. Схема Базы Данных

### `SystemStaffTable`
Связывает глобального пользователя (`User`) с ролью на платформе.
*   `id` (PK)
*   `userId` (FK -> Users)
*   `role` (Enum: SUPER_ADMIN, SUPER_MANAGER, MANAGER)

### `PlatformRequestsTable`
Хранит действия, требующие одобрения (например, активация подписки).
*   `id` (PK)
*   `type` (ACTIVATE_POINT, BLOCK_PARTNER)
*   `status` (PENDING, APPROVED, REJECTED)
*   `requesterId` (Менеджер)
*   `approverId` (Руководитель / Владелец)
    `targetPointId`
*   `payload`: `amount`, `duration`, `isTrial`, `blockReason`

### `PlatformSubscriptionsTable`
Записывает активные/истекшие подписки.
*   `id` (PK)
*   `partnerId`
*   `type` (FIXED_TERM)
*   `startDate`, `endDate`
*   `amount`
*   `isTrial`
*   `isActive`

### `PlatformInvitesTable`
Хранит коды приглашений для присоединения к команде платформы.
*   `code` (PK)
*   `role`
*   `createdBy`
*   `createdAt`

## 3. Бизнес-процессы

### А. Активация (Продажа)
1.  **Менеджер** договаривается с Партнером.
2.  Менеджер заходит в Админ-панель -> Страница Партнера.
3.  Нажимает "Активировать / Продлить".
4.  Заполняет форму: Сумма (напр., 5000 KGS), Срок (напр., 1 Месяц), Тестовый период (Чекбокс).
5.  **Бэкенд**: Создает `PlatformRequest` (Статус: PENDING).
    *   *Валидация*: Если `isTrial` = true, проверяет, использовал ли партнер триал ранее. Если да -> Ошибка.
    *   *Триал*: Всегда фиксированно **14 дней**.
6.  **Руководитель** видит заявку во "Входящих".
7.  Нажимает "Одобрить".
8.  **Бэкенд**:
    *   Обновляет Заявку -> APPROVED.
    *   Создает `PlatformSubscription`.
    *   Обновляет `TradingPoint.isActive = true`.
    *   Обновляет `Partner.status = ACTIVE`.

## 4. API Эндпоинты

### `/platform/requests`
*   `POST`: Создать заявку (Менеджер).
*   `GET`: Список заявок (Фильтр по статусу/создателю).

### `/platform/requests/{id}/approve`
*   `POST`: Выполнить логику активации (Руководитель+).

### `/platform/requests/{id}/reject`
*   `POST`: Отклонить с указанием причины (Руководитель+).

### `/platform/invite`
*   `POST`: Сгенерировать код приглашения.
    *   Владелец может приглашать `SUPER_MANAGER` и `MANAGER`.
    *   Руководитель может приглашать только `MANAGER`.

### `/platform/join`
*   `POST`: Применить код приглашения и получить роль.
